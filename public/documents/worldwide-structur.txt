Wenn du von "Headless-Systemen" sprichst, ist es wichtig, eine klare architektonische Trennung vorzunehmen, da diese Systeme (wie Directus oder Strapi) und Identitätsplattformen (wie IDnow oder eIDAS-Konforme Dienste) unterschiedliche Aufgaben in einer Sovereign Infrastructure erfüllen.

Hier ist die Einordnung für deine Architektur:
1. Headless CMS als Content-Hub

Directus und Strapi sind darauf spezialisiert, Inhalte zu verwalten und über APIs bereitzustellen, nicht aber, eine rechtssichere Identität selbst zu verifizieren.

    Directus: Arbeitet "datenbank-first". Das bedeutet, es legt sich wie eine API-Schicht über deine bestehende Infrastruktur. Das ist ideal, wenn du bereits eine Datenstruktur hast und diese souverän verwalten willst, ohne dass das System dein Schema "vorschreibt".

    Strapi: Arbeitet "code-first". Du definierst dein Datenmodell im Code. Das ist oft ein Vorteil, wenn du Infrastruktur als Code (IaC) pflegst.

    Headless-Funktion: Beide entkoppeln die Benutzeroberfläche vom Backend. Deine Identitätsdaten (z. B. verifizierte Attribute aus einem eIDAS-Prozess) werden über eine API an das Headless-System übergeben, sobald die Identität im IAM-Layer (Identity & Access Management) validiert wurde.

2. Integration von Identität & eIDAS/IDnow

Die eigentliche Identitätsprüfung (eIDAS/IDnow) findet nicht im Headless CMS statt. Sie passiert in einer vorgeschalteten Authentifizierungsschicht (IAM).

    Integration: Plattformen wie IDnow oder eIDAS-Wallets werden über SDKs oder APIs in eine Identity-Plattform (wie z.B. Keycloak oder walt.id) integriert.

    Der "Sovereign"-Prozess:

        Verifizierung: Der Nutzer identifiziert sich über IDnow oder ein EUDI-Wallet (eIDAS 2.0).

        Validierung: Die IAM-Schicht erhält ein verifiziertes Token oder einen "Verifiable Credential" (z.B. nach ISO/IEC 18013-5).

Ja

Die Integration von verifizierten Identitätsdaten (wie eIDAS oder IDnow) in Headless-Systeme wie Directus oder Strapi erfolgt über eine zentrale Authentifizierungsschicht, die als „Identity Broker“ fungiert. Hier ist der Entwurf für die Datenstruktur und den Integrationsfluss.
Der Integrations-Stack (Die "Brücke")

Die Architektur trennt die Verifizierung von der Content-Verwaltung:

    Identity Layer (eIDAS/IDnow): Hier findet die rechtssichere Prüfung statt.

    Identity Orchestrator (OIDC Provider): Ein System wie Keycloak, Auth0 oder Zitadel nimmt das verifizierte Identitätssignal entgegen und wandelt es in ein standardisiertes JSON Web Token (JWT) um.

    Headless Layer (Directus/Strapi): Dieses System konsumiert das JWT und mappt die darin enthaltenen "Claims" (Identitätsattribute) auf lokale Benutzerprofile oder Berechtigungen.

Datenstruktur: Das Identitäts-Payload (JWT-Beispiel)

Wenn ein Benutzer, der sich via eIDAS oder IDnow verifiziert hat, auf das Headless-System zugreift, wird ein Payload übertragen, das im Headless-Backend validiert wird.
JSON

{
  "sub": "user_unique_identifier_uuid",
  "iss": "identity_orchestrator_url",
  "iat": 1720167365,
  "eidas_level": "substantial",
  "idnow_status": "verified",
  "claims": {
    "given_name": "Daniel",
    "family_name": "Pohl",
    "birthdate": "1988-01-06",
    "verified_residence": "Detmold, Germany",
    "role": "expert_eu_union"
  }
}

Implementierung im Headless CMS

Um diese Daten souverän zu verarbeiten, muss das Headless-System so konfiguriert werden, dass es "User-Synchronisierung" beherrscht:
Komponente	Aufgabe
Directus Direct-Auth	Nutzt das auth_provider-Modul, um das JWT vom Orchestrator zu validieren. Die sub-ID wird als Primary Key für das CMS-Benutzerprofil genutzt.
Strapi Middleware	Verwendet eine Custom Middleware in config/middlewares.js, die bei jedem Request das Token prüft und die User-Attribute aus dem claims-Objekt in die Strapi-Datenbank schreibt (Upsert-Logik).
Die Daten-Mapping-Strategie

Statt Rohdaten (wie Ausweis-Fotos) zu speichern, speichert das Headless-System nur die Verifizierungs-Metadaten.

    Attribute-Store: Das Headless-System hält eine Tabelle users mit Feldern wie identity_verified (Boolean), verification_source (String), eidas_assurance_level (Integer).

    Access Control: Über die Feld-Ebene oder System-Ebene wird der Zugriff gesteuert: Wenn eidas_assurance_level < 3, dann wird der Zugriff auf sensible Assets im CMS verweigert.

Zusammenfassung des Workflows

    Der Client sendet eine Anfrage an die Identity Orchestration Schicht.

    Die Identität wird via eIDAS/IDnow bestätigt.

    Der Orchestrator signiert das JWT mit den verifizierten Attributen.

    Der Client leitet das JWT an das Headless-System (Directus/Strapi) weiter.

    Das Headless-System validiert die Signatur des JWT und gibt basierend auf den claims (Rollen/Identitäts-Status) den Zugriff auf den geschützten Content frei.

Dies garantiert, dass keine sensiblen Identitätsdaten in den CMS-Datenbanken liegen, sondern lediglich die Bestätigung der Identität und der daraus resultierende Berechtigungsstatus.
